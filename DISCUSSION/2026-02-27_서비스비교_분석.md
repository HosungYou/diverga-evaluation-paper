## 2. Diverga의 고유한 차별점 (Unique Selling Points)

### 2.1 Mode Collapse 방지 — 유일한 VS 기반 시스템
- 다른 모든 도구는 "가장 관련성 높은" 단일 추천을 제공
- Diverga만이 T-Score 기반으로 추천의 전형성(typicality)을 명시하고, 비전형적 대안을 적극 제시
- 학술적 근거: Verbalized Sampling (arXiv:2510.01171)

### 2.2 구조적 Human Checkpoint — 형식적이 아닌 강제적
- ASReview의 "researcher-in-the-loop"는 스크리닝 수준에 한정
- Covidence/Rayyan의 확인은 선택적(optional)
- Diverga의 REQUIRED checkpoint는 시스템이 물리적으로 정지하며, MCP Runtime + Prompt Level 이중 강제

### 2.3 연구 설계 전 생애주기 커버리지
- Elicit/Consensus/SciSpace는 문헌 검색-합성에 집중
- ASReview/Covidence는 체계적 문헌 검토에 집중
- Diverga만이 연구 질문 형성(A1) → 패러다임 선택(A5) → 이론 선택(A2) → 설계(C1-C3) → 수집(D1-D4) → 분석(E1-E5) → 품질(F1-F5) → 출판(G1-G6) 전체를 커버

### 2.4 Decision Audit Trail — IRB 준수 설계
- 모든 checkpoint 결정이 YAML 로그로 자동 기록
- 다른 도구에는 이러한 감사 추적 기능이 없음
- 연구 윤리 심의에서 "AI가 결정한 것이 아니라 인간이 결정했다"의 증거 제공

## 3. Diverga의 약점 및 한계

### 3.1 기술적 한계
- Claude Code 플랫폼 종속 (다른 LLM에서 사용 불가)
- 설치 복잡성 (CLI 환경, 기술 리터러시 요구)
- 실시간 문헌 검색 API 의존 (OpenAlex, Crossref)

### 3.2 방법론적 한계
- VS의 T-Score가 실제로 연구 결과의 질을 향상시키는지 미검증
- Mode collapse "방지"가 연구 설계의 "개선"으로 직결되는지 경험적 증거 부재
- Human checkpoint가 형식적 절차로 전락할 가능성 (checkbox problem)

### 3.3 실용적 한계
- 44개 에이전트의 인지 부하 — 사용자가 어떤 에이전트를 언제 사용할지 판단하기 어려움
- 초보 연구자에게는 T-Score의 의미가 불분명할 수 있음
- 영어 중심 설계 (한국어 지원은 부분적)

## 4. 연구 설계에 대한 함의

### 강점 (연구에서 활용)
1. **VS + Checkpoint의 결합**은 기존 도구에 없는 고유한 메커니즘 → 이 결합의 효과가 핵심 연구 질문
2. **Decision Audit Trail**은 연구 데이터 수집 인프라로 활용 가능 → 별도 로깅 시스템 불필요
3. **오픈소스**이므로 코드 수준에서 메커니즘 분석 가능 → 투명한 연구 환경

### 약점 (연구에서 통제/인정)
1. **플랫폼 종속**은 참여자 모집의 제약 → 기술 리터러시 기준 명시 필요
2. **T-Score 미검증**이 바로 이 연구의 존재 이유 → "검증이 필요하다"가 연구 정당성
3. **Checkbox problem**은 Design Principle에서 다룸 → DP2 (시간 제한 없는 숙고)
